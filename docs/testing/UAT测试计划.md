# UAT测试计划

## 1. UAT测试目标

验证系统是否满足业务需求,确保系统功能、数据准确性和用户体验符合用户期望,为系统上线提供决策依据。

## 2. UAT测试范围

### 2.1 测试模块

| 模块 | 测试内容 | 优先级 |
|------|---------|--------|
| 企业备案 | 备案信息录入、备案审核 | P0 |
| 数据填报 | 月度填报、数据校验 | P0 |
| 市级审核 | 数据初审、退回 | P0 |
| 省级复核 | 数据复核、汇总 | P0 |
| 数据分析 | 对比分析、趋势分析、取样分析 | P1 |
| 报表导出 | Excel导出、PDF导出 | P1 |
| 系统管理 | 用户管理、权限管理、时限设置 | P1 |
| 密码维护 | 密码修改 | P2 |

### 2.2 测试场景

**场景一: 完整业务流程**
- 企业备案→数据填报→市级初审→省级复核→数据分析→报表导出

**场景二: 审核流程**
- 市级审核通过、市级审核退回、省级复核通过、省级复核退回

**场景三: 异常处理**
- 数据校验失败、超时未报、审核超时

## 3. UAT测试环境

### 3.1 环境配置

| 组件 | 配置 |
|------|------|
| 服务器 | 与生产环境一致 |
| 数据库 | MySQL 8.0,与生产版本一致 |
| 网络 | 独立网络环境 |
| 浏览器 | Chrome 80+,Firefox 75+ |

### 3.2 测试数据

**数据准备:**
- 100家测试企业
- 5个市级管理部门
- 1个省级管理部门
- 6个月历史数据

**数据要求:**
- 使用脱敏的真实业务数据
- 覆盖各种业务场景
- 包含边界值和异常值

## 5. UAT测试用例

### 5.1 测试用例设计原则

- 覆盖所有核心业务流程
- 包含正常场景和异常场景
- 使用真实业务数据
- 验证数据准确性
- 验证用户体验

### 5.2 核心测试用例

#### TC-UAT-001: 企业备案完整流程

**优先级:** P0

**前置条件:** 省级用户已创建企业账号

**测试步骤:**
1. 企业用户使用账号登录系统
2. 进入企业信息→备案信息
3. 填写备案信息:
   - 所属地区: 云南省-昆明市
   - 组织机构代码: A12345678
   - 企业性质: 国有企业
   - 行业: 制造业-电子信息制造业
   - 邮政编码: 650000
   - 联系电话: 13800138000
   - 传真: 0871-12345678
4. 提交备案
5. 省级用户登录系统
6. 进入备案审核
7. 查看企业备案信息
8. 点击"审核通过"

**预期结果:**
- 企业可成功登录
- 备案信息填写正确
- 校验规则生效
- 备案提交成功
- 省级用户可查看备案
- 审核操作成功

#### TC-UAT-002: 数据填报完整流程

**优先级:** P0

**前置条件:** 企业备案已审核通过

**测试步骤:**
1. 企业用户登录
2. 进入数据填报→月度填报
3. 选择调查期: 2026年4月
4. 填写数据:
   - 建档期就业人数: 500
   - 调查期就业人数: 450
   - 减少类型: 经济性裁员
   - 原因说明: 因订单不足,公司进行经济性裁员...
5. 提交数据
6. 市级用户登录
7. 进入数据审核
8. 查看企业填报数据
9. 点击"审核通过"
10. 省级用户登录
11. 进入报表管理
12. 查看市级审核通过数据
13. 点击"复核通过"

**预期结果:**
- 填报页面正常显示
- 减少类型和原因必填
- 数据提交成功
- 市级审核流程正确
- 省级复核流程正确
- 数据状态流转正确

#### TC-UAT-003: 审核退回流程

**优先级:** P0

**测试步骤:**
1. 企业用户提交填报数据
2. 市级用户登录
3. 进入数据审核
4. 查看数据
5. 点击"审核退回"
6. 填写退回原因: "就业人数减少类型未选择"
7. 企业用户登录查看状态
8. 查看退回原因
9. 修改数据
10. 重新提交

**预期结果:**
- 审核退回功能正常
- 退回原因记录完整
- 企业可查看退回状态
- 修改后可重新提交
- 数据状态更新正确

#### TC-UAT-004: 数据分析功能

**优先级:** P1

**测试步骤:**
1. 省级用户登录
2. 进入数据分析→对比分析
3. 选择调查期1: 2026年3月
4. 选择调查期2: 2026年4月
5. 选择对比维度: 地区
6. 点击"生成报表"
7. 查看折线图
8. 查看数据表格
9. 点击"导出Excel"

**预期结果:**
- 对比分析页面正常
- 折线图显示正确
- 数据计算准确
- 表格数据完整
- Excel导出成功

#### TC-UAT-005: 趋势分析功能

**优先级:** P1

**测试步骤:**
1. 省级用户登录
2. 进入数据分析→趋势分析
3. 选择时间范围: 2025年10月-2026年4月
4. 点击"生成趋势图"
5. 查看趋势图
6. 查看趋势数据
7. 验证数据准确性

**预期结果:**
- 趋势图显示正确
- 多个调查期数据对比
- 趋势方向正确
- 数据计算准确

#### TC-UAT-006: 权限控制验证

**优先级:** P1

**测试步骤:**
1. 企业用户登录
2. 尝试访问市级审核功能
3. 尝试访问省级分析功能
4. 市级用户登录
5. 尝试访问省级系统管理
6. 省级用户登录
7. 访问所有功能

**预期结果:**
- 企业用户只能访问本企业功能
- 市级用户只能访问本市功能
- 省级用户可访问全省功能
- 权限控制有效

## 6. UAT测试执行

### 6.1 执行计划

| 阶段 | 时间 | 内容 |
|------|------|------|
| 准备阶段 | 第1天 | 环境部署、测试数据准备、培训 |
| 执行阶段1 | 第2-3天 | 核心业务流程测试(P0用例) |
| 执行阶段2 | 第4-5天 | 功能模块测试(P1用例) |
| 问题修复 | 第6天 | 开发人员修复问题 |
| 回归测试 | 第7天 | 验证问题修复 |

### 6.2 执行要求

**每日执行:**
- 上午: 执行测试用例
- 下午: 执行测试用例、记录缺陷
- 傍晚: 当日测试总结

**缺陷报告:**
- 发现缺陷立即报告
- 填写缺陷报告单
- 附截图和复现步骤

### 6.3 执行记录

**测试执行日志:**
- 日期
- 执行用例数
- 通过用例数
- 失败用例数
- 发现缺陷数
- 执行人员

## 7. UAT验收标准

### 7.1 必须满足条件

- [ ] 所有P0用例100%通过
- [ ] P1用例通过率≥95%
- [ ] 无P0/P1级未修复缺陷
- [ ] 数据准确性验证通过
- [ ] 核心业务流程验证通过

### 7.2 建议满足条件

- [ ] P1用例100%通过
- [ ] P2用例通过率≥90%
- [ ] 用户体验良好
- [ ] 报表格式美观
- [ ] 操作流畅

### 7.3 验收结论

**通过:**
- 满足所有必须条件
- 用户代表签字确认
- 同意上线

**有条件通过:**
- 满足必须条件
- 存在非阻塞性问题
- 限期修复后上线

**不通过:**
- 不满足必须条件
- 存在阻塞性问题
- 修复后重新UAT

## 10. UAT风险控制

### 10.1 风险识别

| 风险 | 可能性 | 影响 | 应对 |
|------|--------|------|------|
| 环境问题 | 中 | 高 | 提前准备,备份环境 |
| 数据问题 | 中 | 高 | 提前准备,多套数据 |
| 人员缺席 | 低 | 中 | 备用人员 |
| 缺陷过多 | 中 | 高 | 提前系统测试,预留修复时间 |

### 10.2 应急预案

**环境故障:**
- 切换备用环境
- 快速修复

**数据错误:**
- 使用备用数据
- 重新准备数据

**人员缺席:**
- 备用人员顶替
- 调整测试计划
