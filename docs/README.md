# 文档管理规范

## 1. 文档管理目标

建立完善的文档管理体系,确保文档完整性、版本可控、易于检索和共享,为项目开发、运维和知识传承提供支撑。

## 3. 文档版本管理

### 3.1 版本号规则

**格式:** 主版本号.次版本号.修订号 (Major.Minor.Patch)

**规则:**
- **主版本号(Major):** 重大变更,如文档结构重组、核心内容重写
- **次版本号(Minor):** 内容增减,如新增章节、补充内容
- **修订号(Patch):** 小修改,如错别字修正、格式调整

**示例:**
- V1.0.0: 初始版本
- V1.1.0: 新增内容
- V1.1.1: 修正错误
- V2.0.0: 重大改版

### 3.2 版本记录

**文档头部格式:**
```markdown
---
title: 需求规格说明书
version: V1.0.0
author: 张三
created: 2026-04-21
updated: 2026-04-21
reviewer: 李四
status: 草稿/评审中/已发布/已归档
---
```

**版本历史:**
```markdown
## 版本历史

| 版本号 | 日期 | 修改内容 | 修改人 | 审核人 |
|-------|------|---------|--------|--------|
| V1.0.0 | 2026-04-21 | 初始版本 | 张三 | 李四 |
| V1.1.0 | 2026-05-01 | 新增数据校验规则 | 张三 | 李四 |
| V1.1.1 | 2026-05-05 | 修正格式错误 | 张三 | - |
```

### 3.3 版本控制流程

```
编写文档
    ↓
自我审核
    ↓
提交评审
    ↓
评审通过
    ↓
发布版本
    ↓
归档旧版本
```

### 3.4 文档状态

| 状态 | 定义 | 操作权限 |
|------|------|---------|
| 草稿 | 编写中,未评审 | 编写人可修改 |
| 评审中 | 提交评审 | 仅评审人可评论 |
| 已发布 | 评审通过,正式发布 | 不可修改,只能创建新版本 |
| 已归档 | 旧版本,已废弃 | 只读 |

## 4. 文档命名规范

### 4.1 文件命名

**规则:**
- 使用小写字母
- 单词用连字符(-)分隔
- 使用有意义的名称
- 避免中文和特殊字符

**示例:**
```
✓ srs.md
✓ architecture-design.md
✓ test-plan.md
✗ SRS.md
✗ 需求规格说明书.md
✗ architecture design.md
```

### 4.2 目录命名

**规则:**
- 使用小写字母
- 使用名词
- 简洁明了

**示例:**
```
✓ requirements/
✓ design/
✓ testing/
✗ Requirements/
✗ req/
✗ docs_about_requirements/
```

## 5. 文档编写规范

### 5.1 Markdown格式规范

**标题:**
```markdown
# 一级标题 (文档标题,每文档一个)
## 二级标题 (章节)
### 三级标题 (小节)
#### 四级标题 (子小节)
```

**列表:**
```markdown
- 无序列表项
- 无序列表项

1. 有序列表项
2. 有序列表项
```

**表格:**
```markdown
| 列1 | 列2 | 列3 |
|----|----|----|
| 内容1 | 内容2 | 内容3 |
```

**代码块:**
````markdown
```java
public class Example {
    // 代码内容
}
```
````

**链接:**
```markdown
[链接文本](链接地址)
```

**图片:**
```markdown
![图片描述](图片路径)
```

### 5.2 文档模板

**通用文档模板:**
```markdown
---
title: 文档标题
version: V1.0.0
author: 作者
created: YYYY-MM-DD
```

## 6. 文档评审流程

### 6.1 评审类型

| 评审类型 | 触发条件 | 参与人 | 输出 |
|---------|---------|--------|------|
| 内部评审 | 文档编写完成 | 编写人、同组成员 | 评审意见 |
| 正式评审 | 内部评审通过 | 项目组、干系人 | 评审纪要 |
| 定期复审 | 文档发布后6个月 | 文档负责人 | 复审报告 |

### 6.2 评审检查单

**内容完整性:**
- [ ] 文档目的明确
- [ ] 范围定义清晰
- [ ] 内容完整无遗漏
- [ ] 图表清晰准确

**格式规范性:**
- [ ] 符合Markdown规范
- [ ] 标题层级正确
- [ ] 表格格式统一
- [ ] 代码块语法正确

**语言准确性:**
- [ ] 无错别字
- [ ] 语法正确
- [ ] 术语使用准确
- [ ] 表达清晰

**版本合规性:**
- [ ] 版本号正确
- [ ] 版本历史完整
- [ ] 元数据完整

## 7. 文档存储与共享

### 7.1 存储位置

**Git仓库:**
- 所有文档纳入Git版本控制
- 使用Markdown格式
- 支持协作编辑
- 版本可追溯

**云盘备份:**
- 定期备份到云盘
- 防止数据丢失
- 支持离线访问

### 7.2 文档检索

**文档索引:**

- Git仓库搜索

## 8. Markdown转换规范

### 8.1 Word转Markdown

**转换工具:**
- Pandoc: `pandoc input.docx -o output.md`
- 在线转换工具
- IDE插件

**转换后检查:**
- 表格格式
- 图片路径
- 列表层级
- 特殊字符

### 8.2 转换注意事项

**表格:**
- 检查单元格合并
- 检查复杂表格
- 必要时简化表格

**图片:**
- 图片提取到独立目录
- 更新图片路径
- 添加图片描述

**格式:**
- 检查字体样式
- 检查颜色标记
- 转换为Markdown等价格式

## 9. 文档维护

### 9.1 维护职责

| 文档类型 | 负责人 | 审核人 |
|---------|--------|--------|
| 需求文档 | 产品经理 | 项目经理 |
| 设计文档 | 架构师 | 技术负责人 |
| 测试文档 | 测试负责人 | 项目经理 |
| 用户文档 | 产品经理 | 业务负责人 |

### 9.2 维护周期

| 文档类型 | 复审周期 | 触发条件 |
|---------|---------|---------|
| 需求文档 | 3个月 | 需求变更 |
| 设计文档 | 6个月 | 架构变更 |
| 测试文档 | 3个月 | 版本发布 |
| 用户文档 | 6个月 | 功能更新 |

### 9.3 文档废弃

**废弃条件:**
- 文档内容已过期
- 被新文档替代
- 相关业务已终止

**废弃流程:**
1. 标记为已废弃
2. 迁移到归档目录
3. 更新文档索引
4. 通知相关人员

## 10. 文档度量

### 10.1 度量指标

| 指标 | 计算方式 | 目标值 |
|------|---------|--------|
| 文档完整率 | 已编写文档/应编写文档 | 100% |
| 文档及时率 | 按时发布文档数/总文档数 | ≥90% |
| 文档评审率 | 已评审文档/总文档数 | 100% |
| 文档缺陷率 | 有错误文档/总文档数 | ≤5% |

### 10.2 文档审计

**审计频率:** 每季度

**审计内容:**
- 文档完整性
- 文档及时性
- 文档质量
- 版本管理规范性

**审计报告:**
- 审计结果
- 发现问题
- 改进建议
